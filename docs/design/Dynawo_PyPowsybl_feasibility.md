# Feasibility assessment: replacing the Dynawo CLI with PyPowsybl

**Status: BLOCKED today by a Dynawo release-cycle mismatch (not architectural).**
powsybl-dynawo pins to official Dynawo *releases* (latest = **1.7.0**); DyCoV runs Dynawo
**1.8.0 development (master)**. So pypowsybl 1.15 generates connections for the 1.7.0
interface, which 1.8.0-master has renamed → **no dynamic simulation completes** here, not
even pypowsybl's own canonical example. A powsybl-dynawo PR for 1.8.0 exists but ships only
**after Dynawo 1.8.0 is officially released**. The catalog gate found earlier is separately
surmountable via `additionalModelsFile`. Last updated 2026-07-08.

**Versions tested:** PyPowsybl **1.15.0** — the latest release on PyPI (no 1.16/2.x;
`pyproject` cap `>=1.7,<2.0` hides no newer release). powsybl-dynawo bundled therein,
Dynawo **1.8.0** (`/opt/Dynawo_v1.8.0_20260701`).

Companion docs: design intent [`Dynawo_PyPowsybl_Backend.md`](Dynawo_PyPowsybl_Backend.md);
reproducible demos [`tools/pypowsybl-vs-native/`](../../tools/pypowsybl-vs-native).

**Code & where it lives (pypowsybl branch):**
- `tools/olf_init_check/network_builder.py` — `PypowsyblNetworkBuilder`: structural
  IIDM only; keyword-only wrappers over `pypowsybl.network.create_*`. Standalone; moved out of the
  `dycov` package (formerly `src/dycov/curves/dynawo/pypowsybl/`) while the integration is undecided,
  and co-located with its only consumer (`compare_init.py`) and test. Targets pypowsybl 1.14 semantics.
- Dependency: optional extra `dynawo-pypowsybl = ["pypowsybl>=1.7,<2.0"]` (`pyproject.toml`).
- Would-be seam: `DynawoSimulator.run_base()` / `run_simple()` (`runtime/dynawo_simulator.py`); retry
  and bisection are backend-agnostic (see the CLI flow in `DyCoV_architecture_map.md`).

> **Note on this assessment's history.** It went through two corrections as evidence
> accumulated: (1) an initial "catalog membership is the gate → not feasible" conclusion,
> then (2) the discovery — from powsybl-dynawo's `base-additional-models` docs — that the
> catalog is **extensible** via `additionalModelsFile`, which re-opens feasibility. The
> current text reflects (2).

## Question

Can DyCoV execute its Dynawo dynamic simulations through PyPowsybl instead of the
Dynawo CLI (`subprocess`), so the CLI backend could be replaced?

## Verdict

**Not proven either way yet, but the catalog is no longer a hard blocker.** PyPowsybl
does not run native DYD/JOBS/CRV — powsybl-dynawo generates them from an IIDM network +
a `ModelMapping`. A model must be known to powsybl-dynawo's model supplier, BUT that
supplier can be **extended at runtime** with `additionalModelsFile`, so non-catalog
models (e.g. `WTG3WeccCurrentSource1`) can be registered and instantiated. Feasibility
now hinges on the items in "Remaining blockers" below, not on catalog membership.

## Key finding: the catalog is runtime-extensible (`additionalModelsFile`)

powsybl-dynawo's base catalog (`models.json`, ~164 models) can be augmented with an
**additional models file** — a JSON registering extra `lib`s **within existing
categories**. In PyPowsybl it is set as a **provider parameter** (config.yaml was NOT
picked up in testing; passing it via `Parameters(provider_parameters=…)` works):

```python
params = dyn.Parameters(start_time=0.0, stop_time=5.0,
    provider_parameters={"additionalModelsFile": "/…/additional_models.json"})
```
```json
{ "WECC": { "libs": [
  {"lib": "WTG3WeccCurrentSource1", "internalModelPrefix": "WTG3", "properties": ["SYNCHRONIZED"]}
] } }
```

Empirical result (ReportNode, WECC_Test WTG3 case):
- Without the additional file → `Model WTG3WeccCurrentSource1 not found for WECC`.
- With it → **`Model WTG3WeccCurrentSource1 … instantiation OK`**.

Constraint (per powsybl-dynawo docs, and verified empirically): additional models
**cannot overload existing ones nor belong to non-existent categories** — only new
*models* inside *existing* categories, never a new category. `add_dynamic_model('IEC', …)`
→ **`No category named IEC`**. Creating a category needs a `ModelConfigLoader` Java SPI
service (an upstream/classpath change), not reachable from PyPowsybl at runtime. The 28
categories include `WECC`, `SynchronousGenerator`, `SynchronizedGenerator`, `Load`,
`Line`, `Transformer`, `InfiniteBus`, … but **no `IEC` category**. Also: additional
models register at run time and do **not** appear in `get_supported_models()`.

## Blocker found: Dynawo interface version drift (2026-07-08)

Attempting a **complete** run exposed a blocker that outranks the model-catalog question
in this environment. Running pypowsybl's own canonical integration example
(`integration_tests/test_dynawo.py`: IEEE-14 + `GeneratorSynchronousThreeWindingsProportionalRegulations`,
which upstream asserts `SUCCESS`) against the installed Dynawo **fails**, always because a
connection variable that powsybl-dynawo 1.15 **generates** does not exist in this Dynawo:

| Element | Variable generated by powsybl-dynawo 1.15 | In Dynawo 1.8.0 |
|---|---|---|
| `ActivePowerVariation` (Step) | `step_step_value` | absent |
| `LoadAlphaBeta` switch-off | `load_switchOffSignal1` / `B3_switchOff` | absent |
| `OmegaRef` ↔ synchronous generator | `omega_grp_0_value` / `generator_omegaPu` | mismatch |

All three installed Dynawo builds are **1.8.0 master** (revs `d03e929`, `94e3823`,
`de0163b`). So **no dynamic simulation completes here — not even the official pypowsybl
example.** This is a systematic **model/connection interface drift** between
powsybl-dynawo 1.15 and Dynawo 1.8.0-master; the earlier per-model failures (WT4B
`omegaRefPu`, WTG3 NPE) are the same root cause, not just incomplete wiring.

Confirmed against Dynawo 1.8.0's model DB (`dynawo/ddb/*.desc.xml` / `*.extvar`): the names
powsybl-dynawo 1.15 generates belong to an earlier Dynawo interface (pre-rename), which
1.8.0 has moved:
- `Step.desc.xml` exposes `step_step` — powsybl-dynawo emits `step_step_value`.
- `GeneratorSynchronousThreeWindingsProportionalRegulations.extvar` needs
  `generator.omegaRefPu`; `DYNModelOmegaRef` exposes `omegaRef_N_value` — powsybl-dynawo
  emits `generator_omegaPu` / `omega_grp_0_value`.

Workaround reach: `additionalModelsFile` can override *model-side* variable names
(`variablePrefix`, `macroStaticRef`), but the builder-internal wiring (OmegaRef↔generator,
event models, network) is generated with fixed names and is **not** overridable from
PyPowsybl — so per-model overrides cannot rescue a realistic case (which needs frequency
coupling + events). The clean fix is version alignment (the Dynawo *release* powsybl-dynawo
1.15 targets, or a powsybl-dynawo that targets 1.8.0). Note the installed builds are 1.8.0
*master* snapshots, not the tagged 1.8.0 release.

Root cause (confirmed with the maintainers' release cycle):
- **powsybl-dynawo pins to official Dynawo *releases*; the latest official Dynawo release
  is 1.7.0.** So pypowsybl 1.15's powsybl-dynawo generates connections for the **Dynawo
  1.7.0** interface — which is exactly the pre-rename naming we observed
  (`step_step_value`, `generator_omegaPu`, `load_switchOffSignal1`).
- **DyCoV runs on a Dynawo 1.8.0 *development* (master) build**, whose model interfaces
  have moved past 1.7.0 → the mismatch.
- There is a **branch/PR to add Dynawo 1.8.0 support to powsybl-dynawo**, but a new
  powsybl-dynawo (and thus pypowsybl) will only be released **after Dynawo 1.8.0 is
  officially released**.
- `VERSION_MIN = 1.5.0` is therefore **necessary but not sufficient**: the interface must
  match the exact release powsybl-dynawo targets (1.7.0 today).

**This is a release-cycle blocker, not an architectural one.** It resolves once (1) Dynawo
1.8.0 ships officially and (2) the matching powsybl-dynawo/pypowsybl is published. Two paths
meanwhile: pin DyCoV to Dynawo 1.7.0 (only if its models/features work there — DyCoV
currently depends on master), or run a dual-Dynawo setup (a 1.7.0 release for pypowsybl,
master for DyCoV's custom models) — which complicates deployment and still leaves the other
blockers (Measurements/PCC wiring, IEC category, numerical parity).

Reproducer: `tools/pypowsybl-vs-native/e2e/` (`run_pypowsybl.py`).

## How powsybl-dynawo works, and why "categories"

powsybl-dynawo is a **file generator + Dynawo launcher**: from the `ModelMapping`
(→ DYD `blackBoxModel`s and their connections), `EventMapping` (→ DYD events),
`OutputVariableMapping` (→ CRV) and the PAR files referenced in config, plus a JOBS built
from `DynamicSimulationParameters`, it writes a working directory (`/tmp/dynawo_<id>/`,
incl. `powsybl_dynawo.xiidm`) and runs `dynawo jobs`. So yes — the DYD/PAR/JOBS/CRV that
DyCoV writes by hand are, in the PyPowsybl path, **generated programmatically** by
powsybl-dynawo.

A **category** is the unit of that generation: not a device type, but a **connection/
wiring contract** with its own *builder*. `get_categories_information()` shows the intent —
`Wecc` = "Grid following WECC", `GridFormingConverter` = grid forming,
`SynchronousGenerator` = "participating in network frequency", `SimplifiedGenerator` =
"not synchronized … fixed frequency", etc. Each builder encodes which terminals/control
variables the model exposes and how they are wired (to the network and to helper models
such as `omegaRef`). WECC wind/PV/BESS therefore share ONE `Wecc` builder; **IEC has no
builder/category** — not because it is a different device, but because IEC 61400-27 models
expose a different interface that no one has implemented a builder for. That is why
`additionalModelsFile` (which only adds *models* to *existing* categories) cannot make IEC
work through a hypothetical `IEC` category.

**Probe (documented, reproducible):** registering an IEC model *under the existing WECC
category* (`probe_iec_under_wecc.json`) reaches `instantiation OK` — model registration is
only a name + prefix lookup and does **not** check the interface. The mismatch surfaces
**later**, at connection generation / Dynawo model build, where the WECC builder wires the
grid-following interface against a model that may not expose those exact variables. So
"instantiates" ≠ "compatible"; the category's *wiring* is the real contract.

**Root cause (decisive):** each powsybl-dynawo builder class hard-codes the exact
macro-connectors / variable names for its category, and `additionalModelsFile` only
supplies `lib` + `internalModelPrefix` + `properties` — it reuses that fixed connector
set. Therefore `additionalModelsFile` is useful **only for models that already conform to
an existing category's exact connector contract** (WECC-family units do; IEC does not).
Making IEC work requires an **IEC builder contributed upstream** to powsybl-dynawo — it
cannot be done from PyPowsybl/config at runtime.

## Remaining blockers (the real open questions)

1. **Explicit wiring that DyCoV builds by hand.** powsybl-dynawo auto-generates
   connections from category templates; it does not accept custom `<connect>` statements.
   The frequency reference (`omegaRefPu`) and the PCC sensing that DyCoV wires via
   `OmegaRef`/`SetPoint` and the `Measurements` ammeter must be reproduced the
   powsybl-dynawo way. In testing, both a catalog model (WT4B → *"omegaRefPu is external
   and not connected"*) and the registered WTG3 (NPE during connection processing) reached
   instantiation but not a completed run, because this wiring was not set up.
2. **No-category constructs.** `Measurements` (source of the PCC compliance signals
   `Measurements_BUS_*`) and the Omega/SetPoint/Ramp helpers have no category and must be
   reconstructed by other means.
3. **Category coverage for IEC.** No `IEC` category exists; whether IEC models can be
   shoe-horned into an existing category with correct wiring is unverified.
4. **Numerical parity.** Even once a case runs, matching the native Dynawo curves
   (`WECC_Test/curves_calculated.csv`) within tolerance is unproven — especially for the
   plant-control tests (e.g. `USetPointStep`), which are sensitive to the sensing point.
5. **The exact case is still not reproducible** (setpoint-step event `Step → WTG3_URefPu`,
   the `Measurements` topology); only an *analogous* case can be built.

## What was verified vs not

- ✅ Version/config compatibility; `additionalModelsFile` registers non-catalog models.
- ✅ WTG3 (non-catalog) instantiates once registered.
- ❌ A completed run producing curves (needs omegaRef/frequency + PCC wiring).
- ❌ Numerical parity against native Dynawo.

## Recommended next steps
1. Build out a full catalog-model case (e.g. WT4B) end-to-end: add the frequency/omegaRef
   model, complete the wiring, run and compare curves to a native Dynawo run.
2. Then repeat with a registered non-catalog model (WTG3) to confirm parity.
3. Resolve the `Measurements`/PCC-sensing reconstruction and the IEC-category question.
4. Keep the Dynawo CLI backend as default/fallback throughout.
